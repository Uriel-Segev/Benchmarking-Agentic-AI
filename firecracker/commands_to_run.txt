================================================================================
FIRECRACKER BENCHMARKING - COMMANDS & WORKFLOW
================================================================================

Scripts:
  firecracker/scripts/    - Firecracker-specific scripts
  run_binary_search.sh    - Primary benchmark (repo root)
  run_campaign.sh         - Multi-task campaign runner (repo root)


--------------------------------------------------------------------------------
SCRIPT OVERVIEW
--------------------------------------------------------------------------------

One-time setup (run once per machine):
  install_firecracker.sh    - Installs firecracker binary, TAP networking, iptables
  setup_base_rootfs.sh      - Creates base Ubuntu 22.04 ext4 rootfs disk image

Building blocks (called automatically, you don't run these directly):
  prepare_task_rootfs.sh    - Installs python/deps into rootfs for a task
  run_task.sh               - Launches a single Firecracker VM
  run_parallel.sh           - Launches N VMs in parallel, collects per-parallel results

Primary benchmark scripts (repo root — use these):
  run_binary_search.sh      - Binary search for max stable parallel VM count,
                              with RAM auto-profiling, crash classification,
                              crash-safe per-probe results, and resume support
  run_campaign.sh           - Runs a cartesian product of tasks × hypervisors
                              sequentially with state tracking and resume support

Legacy scripts (still functional, pre-date binary search):
  run_scaling.sh            - Powers of 2 (1, 2, 4, 8, 16, 32, ...)
  run_scaling_linear.sh     - Linear increments (10, 20, 30, 40, ...)
  run_bottleneck.sh         - Fine-grained sweep around CPU saturation point


--------------------------------------------------------------------------------
STEP 1: MOUNT LARGE DISK
--------------------------------------------------------------------------------

# Check if /mydata is already mounted (profile may have done it automatically)
mountpoint -q /mydata && df -h /mydata || echo "not mounted"

# If not mounted, run this to carve out remaining disk space (~1.8TB on this node):
sudo /usr/local/etc/emulab/mkextrafs.pl /mydata

# Verify — should show ~1.7-1.8T available
df -h /mydata

# Create WORKDIR on the large disk (all rootfs copies will go here)
sudo mkdir -p /mydata/firecracker


--------------------------------------------------------------------------------
STEP 2: MACHINE VERIFICATION
--------------------------------------------------------------------------------

hostname
lsb_release -a || cat /etc/os-release
uname -r
ls -l /dev/kvm


--------------------------------------------------------------------------------
STEP 3: ONE-TIME SETUP
--------------------------------------------------------------------------------

# Make all scripts executable
chmod +x Benchmarking-Agentic-AI/firecracker/scripts/*.sh
chmod +x Benchmarking-Agentic-AI/run_binary_search.sh
chmod +x Benchmarking-Agentic-AI/run_campaign.sh

# Dry run to verify setup
sudo DRY_RUN=1 Benchmarking-Agentic-AI/firecracker/scripts/install_firecracker.sh

# Install firecracker and networking (WORKDIR points to the large disk)
sudo WORKDIR=/mydata/firecracker \
     ROOTFS_URL="https://s3.amazonaws.com/spec.ccfc.min/firecracker-ci/v1.9/$(uname -m)/ubuntu-22.04.ext4" \
     KERNEL_URL="https://s3.amazonaws.com/spec.ccfc.min/firecracker-ci/v1.9/$(uname -m)/vmlinux-5.10.225" \
     Benchmarking-Agentic-AI/firecracker/scripts/install_firecracker.sh

# Verify networking
ip a | grep -E "tap0|192\.168\.100\.1" -n
ip route | head

# Create base rootfs
sudo Benchmarking-Agentic-AI/firecracker/scripts/setup_base_rootfs.sh

# Install jq (used for probe JSON writes and resume)
sudo apt-get update && sudo apt-get install -y jq


--------------------------------------------------------------------------------
STEP 4: PREPARE TASK ROOTFS (optional — run_binary_search.sh does this automatically)
--------------------------------------------------------------------------------

# run_binary_search.sh --task <name> builds the rootfs once before probing.
# Pre-build manually only if you want to reuse the same rootfs across many runs
# without rebuilding it each time.

# hello-world (baseline, no extra deps)
sudo Benchmarking-Agentic-AI/firecracker/scripts/prepare_task_rootfs.sh \
    Benchmarking-Agentic-AI/tasks/hello-world \
    /opt/firecracker/rootfs-hello-world.ext4

# parallelize-compute-squares (no extra deps)
sudo Benchmarking-Agentic-AI/firecracker/scripts/prepare_task_rootfs.sh \
    Benchmarking-Agentic-AI/tasks/parallelize-compute-squares \
    /opt/firecracker/rootfs-parallelize-compute-squares.ext4

# pytorch-model-recovery (installs torch CPU — needs extra disk space)
sudo EXTRA_SPACE=3G Benchmarking-Agentic-AI/firecracker/scripts/prepare_task_rootfs.sh \
    Benchmarking-Agentic-AI/tasks/pytorch-model-recovery \
    /opt/firecracker/rootfs-pytorch-model-recovery.ext4

# hf-model-inference (installs transformers+flask+torch, downloads ~250MB distilbert model)
sudo EXTRA_SPACE=3G Benchmarking-Agentic-AI/firecracker/scripts/prepare_task_rootfs.sh \
    Benchmarking-Agentic-AI/tasks/hf-model-inference \
    /opt/firecracker/rootfs-hf-model-inference.ext4


--------------------------------------------------------------------------------
STEP 5: RUN BENCHMARKS
--------------------------------------------------------------------------------

cd Benchmarking-Agentic-AI/

# --- Single task, single hypervisor ---
# --task builds the rootfs automatically; --label tags the results dir.
# RAM per VM is auto-profiled (16 VMs sampled before binary search begins).

sudo ./run_binary_search.sh \
  --hypervisor firecracker \
  --task hello-world \
  --label 48c \
  --repeats 3 \
  --cooldown 10

# With a pre-built rootfs (skips rootfs prep):
sudo ./run_binary_search.sh \
  --hypervisor firecracker \
  /opt/firecracker/rootfs-parallelize-compute-squares.ext4 \
  --label 48c \
  --repeats 3 \
  --cooldown 10

# Override RAM per VM (skips auto-profiling):
sudo ./run_binary_search.sh \
  --hypervisor firecracker \
  --task hello-world \
  --label 48c \
  --vm-ram 512

# Resume an interrupted run (finds the most recent in-progress run automatically):
sudo ./run_binary_search.sh \
  --hypervisor firecracker \
  --task hello-world \
  --label 48c \
  --resume

# Resume from a specific directory:
sudo ./run_binary_search.sh \
  --hypervisor firecracker \
  --task hello-world \
  --label 48c \
  --resume \
  --output ~/results/hello-world/firecracker/48c/20260301_120000


# --- Multiple tasks / hypervisors (campaign) ---
# Runs every combination of --task × --hypervisor sequentially.
# Run inside tmux for unattended multi-day sessions.

tmux new -s campaign

sudo ./run_campaign.sh \
  --task hello-world \
  --task parallelize-compute-squares \
  --task pytorch-model-recovery \
  --task hf-model-inference \
  --hypervisor firecracker \
  --label 48c \
  --repeats 3 \
  --cooldown 10 \
  --between-runs 60

# Resume an interrupted campaign:
sudo ./run_campaign.sh \
  --task hello-world \
  --task parallelize-compute-squares \
  --hypervisor firecracker \
  --label 48c \
  --resume

--------------------------------------------------------------------------------
TIMING METRICS
--------------------------------------------------------------------------------

Per-instance timing breakdown:
  boot_time_s           - Host launch to guest autorun start
  solution_time_s       - solution.sh execution
  test_time_s           - pytest execution
  task_total_time_s     - Boot-done to pre-shutdown
  shutdown_time_s       - reboot -f to process exit
  total_vm_lifecycle_s  - Host launch to process exit

Aggregate stats (min/max/avg) are computed across all completed instances.


--------------------------------------------------------------------------------
RESULTS STRUCTURE
--------------------------------------------------------------------------------

Each run directory:
  ~/results/<task>/firecracker/<label>/<timestamp>/
    summary.json          top-level metadata; updated after every probe
                          (status: in_progress / completed, lo/hi bounds,
                           max_stable_vms, vm_ram_mb, vm_ram_mb_source, ...)
    probes/
      probe_001_n0001_r1.json   baseline (N=1) probe
      probe_002_n0064_r1.json   binary search probe, repeat 1
      probe_003_n0064_r2.json   binary search probe, repeat 2
      ...
    diagnosis/
      probe_002_n0064/
        dmesg.txt         kernel messages at time of failure
        memory.txt        free -m output
        file_nr.txt       fd usage
        instance_logs/    per-VM stdout/stderr

Each probe file contains: vm_count, stable, exit_code, completed, wall_clock,
  failure_reason (host_oom / guest_oom / vmm_crash / timeout / fd_exhaustion /
                  mmap_limit / partial_completion / unknown),
  timing_summary (boot/solution/shutdown min/max/avg), timestamp.

Campaign directory:
  ~/results/campaigns/<label>/<timestamp>/
    campaign.json         state for all runs (pending/in_progress/completed/failed)
    hello-world_firecracker.log
    parallelize-compute-squares_firecracker.log
    ...


--------------------------------------------------------------------------------
NOTES
--------------------------------------------------------------------------------

- RAM per VM is auto-profiled before the binary search begins (runs 16 VMs,
  measures peak RSS + kernel overhead, allocates ceil(measured * 1.2), min 256MB).
  Override with --vm-ram <mb> to skip profiling.
- Override vCPU count: sudo VCPU_COUNT=1 ./run_binary_search.sh ...
- With 48 cores, expect slowdown around 24+ VMs (48 vCPUs saturated)
- The "pip as root" warning during prepare is harmless (runs inside disposable rootfs)
- Pass the .ext4 rootfs file (not task directory) to avoid re-installing deps every run
- Use tmux for unattended multi-day campaign runs
